# adapt.ts 适配引擎：策略模式与 Pipeline-to-Prompt 翻译

> 跨平台 AI Agent 互操作系统深度解析系列（四）

---

## 适配问题的本质

adapt.ts 要解决的问题是：给定一个 `AgentDescriptor`（从 agent.json 解析出的统一描述），生成目标平台能识别的配置文件。

这看起来是一个简单的格式转换，但实际上涉及三个工程挑战：

1. **格式异构性**：9 个平台的配置格式差异巨大——有的是纯 Markdown，有的需要 YAML frontmatter，有的有特殊的 header 标记
2. **特性不对等**：agent.json 的某些字段在部分平台没有对应概念（如 capabilities、subagents）
3. **安装路径差异**：不同平台的配置文件存放位置完全不同（项目根目录 vs 用户主目录 vs VS Code settings）

adapt.ts 用策略模式 + 模板方法模式解决了这些问题。

---

## 核心函数签名

```typescript
export function adaptAgent(
  agentPath: string,
  target: string,
  targetFile?: string
): AdaptationResult
```

`agentPath` 是 Agent 目录的绝对路径，`target` 是目标平台标识（如 `cursor`、`claude_code`），`targetFile` 是可选的输出文件路径。返回 `AdaptationResult`，包含生成的文件路径和适配详情。

## 策略模式：9 个平台的适配分支

`adaptAgent` 内部通过 `switch(target)` 分发到不同的适配策略。每个分支处理三件事：确定输出路径、转换内容格式、写入文件。

### 输出路径映射

| 目标平台 | 输出路径模式 | 说明 |
|----------|-------------|------|
| `cursor` / `opencode` | `.{target}/commands/{slug}.md` | 项目根目录下的平台子目录 |
| `claude_code` | `.claude/commands/{slug}.md` | 用户主目录的 Claude 配置 |
| `codebuddy` | `.codebuddy/skills/{slug}/SKILL.md` | CodeBuddy 的 Skill 目录 |
| `codebuddy_agent` | `.codebuddy/agents/{slug}.md` | CodeBuddy 的 Agent 目录 |
| `github_copilot` | `.github/agents/{slug}.md` | GitHub 的 Agent 目录 |
| `windsurf` / `trae` | `.{target}/rules/{slug}.md` | Rules 风格目录 |
| `aider` | `CONVENTIONS.md` | 项目根目录，**追加模式** |
| `agents_md` | `AGENTS.md` | 项目根目录，**追加模式** |

`slug` 从 Agent 的 `name` 字段生成，用于构建文件名。追加模式（aider、agents_md）使用 `append: true` 标记，适配器会将内容追加到现有文件末尾而非覆盖。

### 内容格式差异

**Cursor / OpenCode**：纯 Markdown，去除 YAML frontmatter。最简单的格式。

**Claude Code**：纯 Markdown，但支持 Pipeline-to-Prompt 翻译——将 worker.yaml 的步骤序列翻译为自然语言可执行指令。

**CodeBuddy Skill**：YAML frontmatter + HTML 注释标记 + Markdown。frontmatter 包含 name、description 等元数据，HTML 注释用于标记 Skill 的边界。

**CodeBuddy Agent**：纯 Markdown + Pipeline 翻译为提示词。与 Skill 的区别在于，Agent 格式会将 worker.yaml 的 pipeline 步骤翻译为人类可读的执行指令。

**GitHub Copilot**：带 AI agent prompt header 的 Markdown。header 包含 `agent` 标识和触发条件。

## Pipeline-to-Prompt 翻译

这是 adapt.ts 中最有技术含量的功能。当 Agent 包含 `worker.yaml`（即 v3.0 的 subagent 模型）时，适配器需要将 YAML Pipeline 翻译为目标平台能理解的自然语言提示词。

```typescript
function loadWorkerYamlAsPrompt(agentDir: string): string | null {
  const yamlPath = findWorkerYaml(agentDir);
  if (!yamlPath) return null;

  const worker = parseYaml(yamlPath);
  const steps = worker.pipeline || [];

  let prompt = "## Execution Pipeline\n\n";
  for (const step of steps) {
    const toolName = step.tool || step.invoke;
    const toolDesc = toolInstructions[toolName] || toolName;
    prompt += `### Step: ${step.name}\n`;
    prompt += `- Tool: ${toolDesc}\n`;
    if (step.args) {
      prompt += `- Args: ${JSON.stringify(step.args)}\n`;
    }
    prompt += '\n';
  }
  return prompt;
}
```

`toolInstructions` 是一个映射表，将内部工具名翻译为人类可读的描述。例如 `read_file` → "Read file contents"，`llm_chat` → "LLM conversation"，`bash` → "Execute shell command"。

翻译后的提示词被注入到适配后的 Markdown 文件中，让不支持 Pipeline 的平台也能理解 Agent 的执行逻辑——虽然无法真正执行 Pipeline，但至少能让 AI 工具理解 Agent 的工作流程并模拟执行。

## 4 层 fallback 指令加载

当适配器需要获取 Agent 的指令内容时，调用 `parseAgentJson()`，内部实现了一个 4 层 fallback 链：

```typescript
function parseAgentJson(agentDir: string): AgentDescriptor {
  // Layer 1: agent.json 的 instructions 字段
  const agentJson = loadJson(path.join(agentDir, 'agent.json'));
  if (agentJson?.instructions) {
    return parseInstructions(agentJson.instructions);
  }

  // Layer 2: 从 subagents 自动生成
  if (agentJson?.subagents?.length) {
    return generateFromSubagents(agentJson.subagents);
  }

  // Layer 3: SKILL.md（stripFrontmatter 去除 YAML 头）
  const skillMd = readFile(path.join(agentDir, 'SKILL.md'));
  if (skillMd) {
    return { instructions: stripFrontmatter(skillMd) };
  }

  // Layer 4: README.md（最终兜底）
  const readme = readFile(path.join(agentDir, 'README.md'));
  return { instructions: readme || '' };
}
```

外层还有一个文件级 fallback：

```typescript
function loadAgentDescriptor(agentDir: string): AgentDescriptor {
  const agentJsonPath = path.join(agentDir, 'agent.json');
  if (existsSync(agentJsonPath)) {
    return parseAgentJson(agentDir);
  }
  // Fallback: 直接读取 SKILL.md
  const skillMdPath = path.join(agentDir, 'SKILL.md');
  if (existsSync(skillMdPath)) {
    return { instructions: stripFrontmatter(readFile(skillMdPath)) };
  }
  throw new Error(`No agent.json or SKILL.md found in ${agentDir}`);
}
```

两层 fallback 叠加后，系统最多尝试 5 个来源（agent.json instructions → subagents → SKILL.md → README.md → 文件级 SKILL.md fallback），确保任何格式的存量 Agent 都能被识别。

## 导入适配器

adapt.ts 不仅负责"导出"（agent.json → 平台格式），还负责"导入"（平台格式 → agent.json）。导入适配器有 7 个：

| 适配器 | 源文件 | 解析逻辑 |
|--------|--------|----------|
| `cursor-import.ts` | `.cursorrules` | 纯 Markdown，直接作为 instructions |
| `claude-import.ts` | `CLAUDE.md` / skill 文件 | 提取 Markdown 内容，去除 frontmatter |
| `codebuddy-import.ts` | CodeBuddy 配置 | 解析 YAML frontmatter + Markdown body |
| `github-import.ts` | Copilot instructions | 提取 AI agent prompt header 后的 Markdown |
| `vscode-import.ts` | VS Code 配置 | 解析 settings.json 中的 agent 相关字段 |
| `jetbrains-import.ts` | JetBrains 配置 | 解析 XML 格式的配置文件 |
| `openai-gpts-import.ts` | OpenAI GPTs JSON | 解析 GPTs 的 instructions 和 tools 定义 |

每个导入适配器的输出格式统一为 `AgentDescriptor`，后续由序列化器转换为标准 agent.json。

## 适配结果的数据结构

```typescript
interface AdaptationResult {
  target: string;           // 目标平台标识
  files: AdaptedFile[];     // 生成的文件列表
  warnings: string[];       // 降级警告（如"capabilities 不被支持"）
  skipped: string[];         // 跳过的字段
}

interface AdaptedFile {
  path: string;             // 输出文件路径
  content: string;          // 文件内容
  append: boolean;          // 是否追加模式
}
```

`warnings` 数组记录了适配过程中的降级信息，帮助用户了解哪些特性在目标平台上不可用。这对于调试"为什么 Agent 在 Cursor 上能用但在 Claude Code 上效果不同"非常有帮助。

---

## 设计模式总结

adapt.ts 的设计体现了几个经典模式的应用：

- **策略模式**：`switch(target)` 分发到不同适配策略，新增平台只需添加一个 case 分支
- **模板方法模式**：`loadAgentDescriptor → parseAgentJson → 4 层 fallback` 定义了指令加载的骨架，具体解析逻辑由各层实现
- **适配器模式（GoF）**：将 AgentDescriptor 的统一接口转换为各平台的具体格式
- **Pipeline-to-Prompt 翻译**：将结构化的 YAML Pipeline 反序列化为自然语言，是一种"降维适配"策略
